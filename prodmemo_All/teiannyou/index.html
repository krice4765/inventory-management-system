2. 現在のシステム概要（非技術者向け説明）

システムの仕組み

現在のシステムは、クラウド上のデータベース（インターネット上にあるデータ保管庫）に商品情報や在庫情報を保存し、Webブラウザ（Internet ExplorerやChromeなど）で操作する仕組みとなっております。

現在利用可能な機能一覧

機能分類 機能名 概要
商品管理 商品マスター管理 図面番号（DRW-A001-2025等）による商品登録・編集
材質・仕様管理 アルミニウム、真鍮、銅合金等の材質情報管理
価格管理 標準価格・販売価格の一元管理
仕入先管理 仕入先マスター 品質等級（A+、A、B+等）による仕入先分類
支払条件管理 月末締め翌月末払い等の条件管理
在庫管理 在庫数量管理 現在庫・最小在庫レベル・発注点の管理
在庫アラート 発注点を下回った商品の自動警告
発注管理 発注書作成 仕入先別の発注書作成・管理
統計機能 ダッシュボード 在庫状況・発注状況の可視化

技術的基盤の安定性・拡張性

  • 安定性：Cloudflare（世界的なクラウドサービス）により高い可用性を実現
  • 拡張性：PostgreSQL（企業向けデータベース）により大容量データにも対応可能
  • セキュリティ：認証システムによりアクセス制御が実装済み
  • バックアップ：データベースの自動バックアップが定期実行

アクセス方法と利用環境

  • アクセス方法：Webブラウザで専用URLにアクセス
  • 対応ブラウザ：Chrome、Edge、Firefox、Safari
  • 対応デバイス：Windows PC、Mac、タブレット、スマートフォン
  • インターネット環境：常時接続が必要

3. 富士精工様要望の整理と課題分析

図面管理・PDF貼付の業務フロー要件

現在の業務フロー（想定）

  1. 発注が必要な商品を特定
  2. 該当商品の図面番号を確認
  3. 図面PDFファイルを手動で検索・取得
  4. 発注書を作成
  5. 発注書と図面PDFを手動で結合または別添付
  6. 仕入先へ送付

理想の業務フロー（要望）

  1. 発注が必要な商品を特定
  2. システムで図面番号を入力
  3. システムが自動で図面PDFを検索・取得
  4. システムが発注書PDFに図面PDFを自動貼付
  5. 完成した発注書PDFをダウンロードして仕入先へ送付

Access活用の意図と現状の課題

項目 Accessの利点 現状の課題
操作性 慣れ親しんだインターフェース Webシステムとのデータ同期が困難
カスタマイズ性 柔軟な帳票作成・PDF生成が可能 データの一元管理ができていない
データ管理 ローカルでの高速処理 複数ユーザーでの同時利用に制限
将来性 既存業務フローとの親和性 他社展開時のスケーラビリティ不足

データ統合の必要性と期待効果

  • データの一元化：AccessとWebシステムの重複データを統合し、単一の正確なデータソースを実現
  • リアルタイム同期：在庫数量や発注情報の即座な反映
  • 業務効率化：データ入力作業の重複を排除
  • データ精度向上：手動転記によるミスを防止

他社展開時の考慮事項

  • スケーラビリティ：複数拠点・多数ユーザーでの同時利用対応
  • 標準化：各社固有の業務フローに依存しない汎用性
  • 保守性：システム管理・アップデートの一元化
  • セキュリティ：企業間でのデータ分離とアクセス制御

アプローチC: ハイブリッド型（推奨）

概要

WebシステムとAccessの両方を活用し、段階的に機能を統合していくアプローチです。PDF管理はWeb側で行い、Accessからは必要に応じてアクセスできる柔軟な構成です。

段階的導入のメリット

  • リスク分散：段階的導入によりリスクを最小化
  • 継続性：既存業務を停止せずに新機能を追加
  • 学習効率：ユーザーが段階的にシステムに慣れることが可能
  • 投資効率：効果を確認しながら段階的に投資

長期的な発展性

  • 拡張性：将来的にWeb完全統合への移行が容易
  • 柔軟性：他社展開時にAccess部分の調整が可能
  • 互換性：既存システムとの共存が可能

推奨理由

  1. 現実的：既存資産を活用しながら新機能を実現
  2. 段階的：リスクを管理しながら確実に効果を実現
  3. 将来性：他社展開を見据えた設計が可能
  4. 費用対効果：最も効率的な投資配分が可能

5. 推奨開発プラン

推奨アプローチとその理由

ハイブリッド型アプローチを強く推奨いたします。以下の理由により、貴社にとって最適な選択と考えております：

推奨理由の詳細

  1. 既存投資の最大活用：AccessやWebシステムへの既存投資を無駄にすることなく、それぞれの長所を活かした統合が可能です。
  2. 段階的リスク管理：一度に大きな変更を行うのではなく、段階的に機能を追加することで、業務への影響を最小限に抑えられます。
  3. 将来性の確保：他社展開時にはWeb側の機能を拡張し、Access依存度を段階的に下げることが可能です。
  4. 現実的な導入：ユーザーの慣れ親しんだAccessインターフェースを維持しながら、最新のWeb技術の恩恵を受けられます。

6. 機能詳細と技術仕様

PDF生成・結合機能の詳細

機能名 概要 技術仕様
図面PDFアップロード 図面番号に紐付けてPDFファイルを保管 Webブラウザからのドラッグ&ドロップアップロード対応
発注書PDF生成 発注情報から標準的な発注書PDFを自動生成 テンプレート方式により柔軟なレイアウト対応
PDF自動結合 発注書PDFと図面PDFを自動的に結合 図面番号による自動検索・結合処理
PDF検索・管理 図面番号、商品名等によるPDFファイル検索 高速全文検索、プレビュー表示機能

図面管理システムの仕様

  • 対応ファイル形式：PDF（推奨）、JPG、PNG
  • ファイルサイズ制限：1ファイルあたり最大10MB
  • 保管容量：初期設定100GB（拡張可能）
  • バックアップ：自動バックアップ（毎日）、世代管理（30世代）
  • アクセス制御：ユーザー権限による閲覧・編集制限
  • 検索機能：図面番号、商品名、アップロード日時による検索

Access-Web連携の技術的実装方法

  1. ODBC接続設定：
    • PostgreSQL ODBCドライバーのインストール
    • 接続文字列の設定（暗号化通信対応）
    • 認証情報の安全な管理
  2. データ同期機能：
    • リアルタイム同期（在庫数量等の重要データ）
    • バッチ同期（マスターデータ等の更新頻度が低いデータ）
    • 競合解決機能（同時更新時の処理）
  3. エラーハンドリング：
    • 接続断時の自動再接続機能
    • データ更新失敗時のロールバック処理
    • エラーログの自動記録・通知

データ移行計画の具体的手順

  1. データ分析・設計（1週間）
    • 既存Accessデータの構造分析
    • Webシステムとのマッピング設計
    • データクレンジングルールの策定
  2. 移行ツール開発（2週間）
    • データ抽出・変換・投入ツールの開発
    • データ整合性チェック機能の実装
    • エラーハンドリング機能の実装
  3. テスト移行・検証（1週間）
    • テスト環境でのデータ移行実行
    • データ整合性の確認
    • 機能動作テスト
  4. 本番移行・運用開始（1週間）
    • 本番環境でのデータ移行実行
    • 最終確認・動作テスト
    • 運用開始・ユーザートレーニング

8. リスク分析と対策

技術的リスク

リスク項目 リスクレベル 影響 対策
ODBC接続の不安定性 中 Access-Web間のデータ同期に遅延や失敗が発生 冗長接続設定、自動再接続機能、手動同期バックアップ機能の実装
データ移行時の整合性エラー 中 データの不整合により業務に支障 段階的移行、詳細なテスト計画、ロールバック手順の準備
PDF処理のパフォーマンス問題 低 大容量PDFファイルの処理で遅延発生 ファイルサイズ制限、非同期処理、プログレス表示の実装
システム間の認証連携エラー 低 Access側からWebシステムへのアクセス拒否 専用認証トークンの発行、定期的な認証更新機能

運用リスク

リスク項目 リスクレベル 影響 対策
ユーザーの操作ミス増加 中 新機能への不慣れによる作業効率低下 段階的トレーニング、操作マニュアル作成、サポート体制強化
既存業務フローとの不整合 中 業務の混乱、効率低下 業務フロー分析、カスタマイズ対応、段階的導入
システム依存度の増加 低 システム障害時の業務停止リスク 手動代替手順の整備、定期的なバックアップ、冗長化対応

各リスクの対策と軽減方法

技術的リスク対策

  • 開発段階での対策：プロトタイプ開発による事前検証、段階的開発によるリスク分散
  • テスト段階での対策：包括的なテストシナリオ、負荷テスト、障害復旧テスト
  • 運用段階での対策：24時間監視体制、自動アラート機能、迅速な障害対応体制

運用リスク対策

  • 導入前対策：ユーザーヒアリング、業務フロー分析、カスタマイズ要件定義
  • 導入時対策：段階的ロールアウト、集中トレーニング、専任サポート配置
  • 導入後対策：定期的なフォローアップ、改善提案、継続的なサポート提供

技術用語 一般語での解説
PostgreSQL 企業向けデータベースソフトウェア。大量のデータを安全に保管・管理できる
ODBC接続 AccessからWebシステムのデータベースに接続するための仕組み
PDF結合 複数のPDFファイルを1つのファイルにまとめる機能
API連携 システム同士が自動的にデータをやり取りする仕組み
クラウドストレージ インターネット上にファイルを保管できるサービス
Webブラウザ インターネットを閲覧するためのソフト（Chrome、Edge等）
データ同期 複数のシステム間でデータを最新状態に保つ処理
バックアップ データの紛失に備えてコピーを保管すること

前提条件・要確認事項リスト

技術的前提条件

  • Microsoft Access 2016以降がインストールされていること
  • PostgreSQL ODBCドライバーのインストールが可能であること
  • インターネット接続が常時利用可能であること
  • Windows 10以降のOSが使用されていること

業務的前提条件

  • 図面番号による商品管理が現在も実施されていること
  • 図面PDFファイルがデジタル形式で存在すること
  • 発注業務が月間100件以上発生していること
  • 複数ユーザーでの同時利用が必要であること

要確認事項

  • 現在の図面PDFファイルの総容量と件数
  • Accessファイルの現在のサイズと構造
  • 移行対象データの詳細範囲
  • ユーザー数と同時利用数の想定
  • セキュリティ・権限管理の要件
  • 他システムとの連携要件
